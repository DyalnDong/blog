0226 - 0305 本周回顾
过去一周，几乎把所有空闲时间都砸在了 OpenClaw 上，甚至开始感觉 AI 的出现，并没有减少工作量，反而加长了工作的时间。
最开始的想法，只是想把散乱的人生，用龙虾给规整起来，给自己配一只常驻的 AI 助理，接管生活的放方面，但现实是另外一回事。
即使用了最强的模型，最多的工具，广泛搜集的插件，问题也是如同打地鼠一样层出不穷：成本失控、上下文膨胀、记忆混乱、重启断流、工具调用卡死……
这周基本就是在这些问题里沉沦，就很像是在给一台刚组好的机器装底盘、装刹车、装仪表盘等等，每个东西都要自己去拼接搭建。从这个角度去看，OpenClaw 要想真正的变成生产力工具，关键不是换更强的模型，模型只是发动机，真正决定这台车能不能长期开下去的，是在使用的人。
最大的坑：
我一直都很喜欢用最强的模型去解决问题，所以一上来，就配置了 Claude Opus 4.6，模型本身不是问题，但问题是早晨起来看了眼账单，一晚上差不多烧了 200 美金，我心想，这也没有做什么事情呀，这个模型有贵的这么夸张吗。
后来仔细复盘才发现，是配置链路搞错了。当时在配置一个工具（Firecrawl）时，一次 JSON 重写把
openclaw.json
意外覆盖了。结果就出现了三件连锁反应：
全局默认模型从
Kimi K2.5
变成
Claude Opus 4.6
cron 任务和日常会话都没有显式指定模型
所有请求全部继承默认模型
然后再叠加一个隐藏问题：
memory-sync
任务是
sessionTarget=main
也就是直接注入主会话。于是到这里就会看到一个非常迷惑的现象：表面上没多少对话，但 token 在疯狂上涨。
让 AI 帮忙来拆解账单之后，发现最主要的消耗来自三类任务。第一就是 memory-sync cron 这个东西，之前设定是每天自动跑一次，会读取大量 memory 文件，默认模型改了 Opus 后，单次成本就在  $10–15，而且最麻烦的是很难单独去做审计；
第二类的消耗就是日常交互会话，很多时候我可能只是问一句 “帮我总结一下这个网页”，但实际发生的是网页全文被抓取，几十 KB 内容直接进入上下文，然后模型带着整篇文章对话，成本自动就暴涨。
第三类的消耗则是 cache write（很多人忽略），Opus 的 cache write 成本非常高。我曾经看到过这样一条记录：1.2M cached tokens cache cost ≈ $7.50，而这只是缓存费用。
所以优化方面就是对模型就行了分层处理，主力模型切换成了 Claude Sonnet，日常的任务稳定、工具调用成功率高、成本可控，而更便宜的任务，则交给了 Kimi K2.5，中文长文处理性价比很好，而关键设计、架构收敛、最终审查时使用就交给了 Opus 4.6。
上下文失控
成本控制住之后，很快遇到第二堵墙，上下文失控。因为 OpenClaw 的工具链很方便，方便到你会下意识让它尽量多去抓取，多带一点上下文，结果就是
web_fetch
抓网页全文，邮件日报喂进四个邮箱，启动时候还加载一堆记忆文件。导致的结果就是模型化了巨量时间在垃圾堆里找线索。
所以记忆方面，在上周基础上，又把长期记忆拆成三层：P0，硬规则 / 身份 / 边界，这个要始终加载；P1，最近 7 天活跃信息，这个滚动加载；P2，历史归档 只做语义检索召回，这个的核心逻辑是：让模型先轻装上阵，需要时再召回历史。
另外的话，就是上线了语义搜索，把历史记忆全部做了向量化。使用：Qdrant + embedding + BM25 混合检索，效果还是很明显的，过去是把几万 token 历史硬塞进上下文，现在是先检索，再精简注入。
量化一下的话，那就是启动上下文从 8.5k tokens 降低到了 4.3k tokens，将近 50% 的减少，后续需要历史信息时，一般只额外加载 1–2k tokens，这样对成本和稳定性都有帮助。
稳定性其实比聪明更重要
这周还有一类问题，不是模型出的问题，是系统工程出现了问题。某次重启电脑后，OpenClaw 完全不回复。第一反应是就是 CodeX 把系统又搞坏了。结果排查发现：Gateway 正常运行但有 session 卡住，状态是
missing tool result
。清理卡住的 session 重启 gateway后，系统立刻就恢复了。
然后就是 stream_read_error，各个聊天渠道都出现过，最后确认原因是上游 provider 的流式输出断了，然后 OpenClaw 就把错误码原样吐出来。然后花了巨量的时间去修复，始终无法修复，最后尝试换了 API 提供商，好像就不怎么出现了，咸鱼出品多少还是不稳定的。。
踩坑后也有产出
虽然前面踩了很多坑，但也不是完全没有效果，至少让 OpenClaw 能够多少拥有点自我进化的能力。比如给它加了一个自检系统，核心思路是不要让同一个坑踩两次。
于是就设定了下面的流程，任务开始前，查询历史纠错经验 — 自动注入上下文 — 如果我纠正它，就进入待确认列表 — 写入规则库，这个能解决一个多会话、多智能体环境下经验无法继承的问题。
第二就是 Skill 路由套娃机制，让 Skill 可以调用 Skill。这就意味着：复杂任务可以被拆成可复用链路。例如：检索记忆 → 读取网页 → 总结 → 写入知识库，这个不用每次重新拼流程，直接走就可以了。
第三就是学着推友，复刻了 Eat 吞噬系统，这个是比较有意思的模块。比如给龙虾一个文章 URL，它会识别能力，然后拆解结构生成 Skill 模板。这些 Skill 在我确认之前不会变成正式技能，会先进入 staging 等待审批。这意味着：OpenClaw 能够自己去拓展自己，像颗树一样去成长。
其他的，继续去修 Bug 吧，周末愉快。
